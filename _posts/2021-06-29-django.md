## 상태코드
상태 코드

* 1XX : Informational(정보제공) → 임시적인 응답(response)으로, 현재 클라이언트의 요청(request)까지 처리되었으니 계속 진행하라는 의미

* 2XX : Success(성공) → 클라이언트가 요청(request)한 동작을 서버에서 성공적으로 처리되었다는 의미
  * <200 OK 성공> → 서버가 요청을 성공적으로 처리

  * <201 Created 생성됨> → 요청이 처리되어 새로운 리소스가 생성

  * <202 Accepted 허용됨> → 요청은 접수했지만 처리가 완료되지 않음

* 3XX : Redirection(리다이렉션) →클라이언트는 요청을 마치기 위해 추가적인 동작을 해줘야함, 주로 서버의 주소 or 요청한 url의 웹 문서가 이동되었으니, 그 주소로 다시 시도해보라는 의미
  * <301 Moved Permanently 영구 이동> →지정한 리소스가 새로운 url로 이동했다.

  * <303 See Other 다른 위치 보기> →다른 위치로 요청하라.

  * <307 Temporary Redirect 임시 리다이렉션> →임시로 리다이렉션 요청이 필요하다.

* 4XX : Client Error(클라이언트 에러) → 없는 페이지를 요청하는 것처럼 클라이언트의 요청(request)이 잘못된 경우 즉 클라이언트 요청에 오류가 있다.
  * <400 Ban Request 잘못된 요청> →요청이 잘못됨.

  * <401 Unauthorized 권한 없음> →지정한 리소스에 대한 액세스 권한이 없다..

  * <400 Forbidden 금지됨> →지정한 리소스에 대한 액세스가 금지됨.

  * <404 Not Found 찾을 수 없음> → 지정한 리소스를 찾을 수 없다.

* 5XX : Server Error(서버 에러) → 클라이언트 요청은 유효한데 서버 측에서 메시지 처리에 문제가 발생한 경우. 서버의 부하, DB 처리에서 오류 등등
  *  <500 Internal Server Error 내부 서버 오류> →  서버쪽에서 에러가 발생했다.

  * <502 Bad Gateway 불량 게이트웨이> → 게이트웨이 or 프록시 역할을 하는 서버가 그 뒷단의 서버로부터 잘못된 응답을 받았다.

  * <503 Service Unavailalbe 서비스 제공 불가> → 현재 서버에서 서비스를 제공할 수 없음.

## MVT패턴

웹 프로그램 개발 시 일반적으로 MVC(Model-View-Controller)패턴을 사용

Model : 데이터, View : 사용자 인터페이스,  Controller : 데이터를 처리하는 로직 → 이렇게 구분하여 서로 영향을 주지 않도록 설계함.

장고에서는 MVC 패턴에서 View를 Template,  Controller를 View라고 표현하여 MVT(Model - View - Template)

* Model : DB에 저장돼있는 데이터를 가져오거나 CRUD
* Template : 사용자에게 보여지는 UI
* View : 프로그램 로직이 동작하여 데이터 가져오고 Template에게 전달함, 버튼을 눌렀을 때 어떤 함수를 호출하며 데이터를 어떻게 가공할 것인지 결정
![](https://siiiido.github.io/assets/images/django.png)


MVT과정

- 클라이언트로부터 요청(request)을 받으면 URLconf를 이용하여 URL 분석(URLconf : 요청에 들어있는 URL이 url.py 파일에 정의된 url패턴과 매칭되는지 분석)

- URL 분석 결과를 통해 URL에 대한 처리를 담당할 VIEW를 결정
- VIEW는 DB처리가 필요하면 MODEL을 통해 처리하여 결과 받음
- VIEW가 할 일이 끝나면 TEMPLATE을 사용하여 클라이언트에 전송할 HTML 파일 생성
- VIEW는 최종 결과로 HTML 파일을 클라이언트에게 보내서 응답(request)

MVT  코딩 순서

1. 프로젝트 뼈대 만들기 : 프로젝트 및 앱 개발에 필요한 디렉토리와 파일 생성
2. 모델 코딩하기 : 테이블 관련(db) 사항을 개발(model.py, admin.py 파일)
3. URLconf 코딩하기 : URL 및 뷰 매핑 관계를 정의(urls.py 파일)
4. 템플릿 코딩하기 : 화면 UI 개발(templates/ 하위 파일들)
5. 뷰 코딩하기 : 애플리케이션 로직 개발(views.py 파일)

## 템플릿 변수 : {{ variable }}

예를 들어 foo.bar라는 템플릿 변수가 있으면

- foo가 사전 타입인지를 확인 → 그렇다면 foo['bar']로 해석
- foo의 속성을 찾음 → bar라는 속성이 있으면 foo.bar로 해석
- foo가 리스트인지를 확인 → 그렇다면 foo[bar]로 해석

템플릿 시스템은 정의가 되어 있지 않은 변수를 사용할 때 빈 문자열(' ')로 채워주고, 이 값을 변경할려면 settings.py파일에서 TEMPLATE_STRING_IF_INVALID 라고 속성을 지정해주면 된다. 

디폴트는 빈 문자열임

## from
action : form 데이터를 어디로 보낼지 지정해줌

method : 어떤 HTTP 메소드(GET, POST)로 보낼지 지정해줌 → 장고에서는 form 데이터를 전송할 때 POST방식만 사용함

장고는 form 처리를 위해 3가지 기능 제공

1. form 생성에 필요한 데이터를 form 클래스로 구조화함
2. form 클래스의 데이터를 렌더링하여 HTML form 만들기
3. 사용자로부터 제출된 form과 데이터를 수신하고 처리함

장고에서 객체 렌더링할 때 3가지 과정으로 이루어짐

1. 렌더링할 객체를 view로 가져오기(ex] 데이터베이스로부터 객체를 추출하기)
2. 그 객체를 템플릿 시스템으로 넘겨주기
3. 템플릿 문법을 처리해서 HTML 마크업 언어로 변환하기

unbound form : 데이터가 없는 폼 → 렌더링되어 사용자에게 보여질 때 비어있거나 디폴트 값으로 채워짐

bound form : 제출된 데이터를 갖고 있어서 데이터의 유효성을 검사하는데 사용
view에서 form 처리

처음 사용자에게 보여주는 form과 사용자가 데이터를 입력한 후 제출된 form을 구분하여 처리해야함

- view가 GET 방식으로 요청을 받은 경우에는 처음 사용자에게 보여주는 form
- view가 POST 방식으로 요청을 받은 경우 사용자가 데이터를 입력한 후 제출된 form
